# 思考学（しこうがく） ─ どう考えたのかを残す（思考の再現レシピ）

> 七つの蔵：**十四式＝どう作るのか／遊戯学＝なぜ作るのか／設計学＝どの高さ・どの順で作るのか／畜史＝何を作ったのか／思考学＝どう考えたのか／肥料学＝なぜ終えたのか／技術学＝何で作るのか。**
> 畜史が「成りしもの」を録し、設計学が「降りる順」を示すなら、思考学は**判断が分かれた場所で、どちらへ・なぜ曲がったか**を録す。
> 目的はただ一つ：**軽量な頭（小さなAIモデル・急いでいる人・記憶を持たない次のこっこ）でも、この手順をなぞれば同じ質の仕事を再現できる**こと。
> 公開ページは `shikogaku/index.html`（蔵書 `/Library/` の棚）。内容を変えたら両方そろえること。
>
> 起：丙午2026・文月二、島ぐらし2（`shima2/`）を建てた思考の記録より結ぶ。

---

## 巻一 ─ 標準手順（新作・大改修の再現レシピ）

**順番を守ること。飛ばした段が、あとで手戻りになる。**

1. **蔵を読む**（森の地図）… CLAUDE.md → `docs/畜学.md`（七相＋運用の鍵）→ `docs/設計学.md`（森林木）→ `docs/遊戯学.md` → 十四式（`jushishiki/index.html`）。ラベルで済ませず開いて読む。
2. **先行作を批判的にレビューする** … 対象（前作・類似作）のコードを実際に開き、①描画は何で作られているか ②遊びの動詞は何か ③畜学七相に照らして違反はどれか、を**忖度なしに**書き出す。ほめるためではなく、次作の設計材料を採るために読む。
3. **畜史で「死んだ木」を調べる** … `changelog/index.html` を `end` タグで検索。**同じ名前・同じ企画が過去に死んでいないか**。死んでいたら `docs/肥料学.md`（死因の記録）を読み、その死因（例：mori2＝Three.js外部依存・iOSシェーダ精度）を必ず設計制約に繰り入れる。
4. **森→林→木で降りる**（設計学）… 森＝畜物語の掟と共有臓器（七相／`rankings/*` 祭壇／🔄強制更新／ブラックこっこ／1ファイル）。林＝この木が連なる束（例：島ぐらしの林＝TPS島・採集・逆縁）。木＝今回の一作。**木の仕様を書く前に、森と林への接続を一行ずつ書く。**
5. **七相との衝突を先に解く** … ジャンルの性質と掟がぶつかる場所を実装前に決める。例：島ぐらしは本来「終わらない」ジャンル → 弐（焔の形）の核は**60秒という数字ではなく「底（必ず終わる）」**。ゆえに「7日間で必ず終わる島ぐらし」に変換する。**掟は捨てない、ジャンルの方を曲げる。**
6. **投資の優先順位を数で決める** … 「すごいグラフィック」は総量でなく順序。効果の大きい順：**①太陽の影（シャドウマップ）②ビルボードを本物のメッシュに ③水の深さ・岸の泡 ④ポスト（ブルーム等）**。①と②だけで絵は別物になる。逆に細部（草の1本の形など）から始めない（＝木から積むな）。
7. **全部にフォールバックを敷く** … シェーダは `tryProg`（失敗したら基本版）、FBOは `checkFramebufferStatus`、機能フラグ（SHADOW_OK/POST_OK）で分岐。**GLSLの精度は頂点・断片ともhighpで統一**（mori2はここでiOSに殺された）。重い端末は 草→影→ポスト の順に自動で落とす。
8. **実際に遊んで検証する**（構文チェックだけで済ませない）…
   - `node --check` を全インライン`<script>`に。
   - ローカルサーバ＋ヘッドレスブラウザ（Playwright＋Chromium）で**起動→開始→終端まで**を通し、`pageerror` ゼロとスクリーンショットの目視を確認。
   - ゲームが長い場合に備え、**検証用フック**（`window.__ゲーム名` に state取得・フェーズスキップ・タイムスケール）を仕込んでおく（先例：`rescue/` の `window.__rescue`）。
9. **共有臓器を繋ぐ** … 🔄強制更新ボタン（雛形 `poko/` の reloadBtn／タイトル右下）、音の番人（pagehideでAudioContextをclose）、Firebase非同期ローダー＋`rankings/<名前>`（新規DB不要）、トップ目録 `index.html`・`sitemap.xml`・`sw.js` のCACHE版上げ。
10. **継として還す** … 畜史にエントリを追加 → 判断が分かれた場所を**思考学（この書）に追記** → コミット＆push → mainへPR作成 → **マージはユーザーに委ねる**（自分で押さない）。

## 巻二 ─ 判断の記録：島ぐらし2「ななつの夜明け」の場合

実際に分かれ道でどう考えたかの記録。同じ形の迷いに出会ったら、まずここを読む。

### 一・前作（mori）レビューの視点
- 「PCゲーマーの目」で見ると、moriの弱点は**世界のすべてが紙（カメラを向くビルボード絵文字）**であること・**太陽があるのに影が落ちない**こと。器用なシェーダ（ACES・大気・フレネル）より先に、この二つが没入を壊していた。
- 遊びの側は「システムの足し算」（わざ・そらわざ・月・諸島・ジム…）で幅はあるが、**底（終わり）が無い**＝畜学弐に反し、総資産ランキングは**時間を注いだ者が勝つ祭壇**＝四の本義（同時に顔を上げる）からズレていた。
- 教訓：**レビューは「足りない機能」でなく「掟との衝突」と「没入を壊す最大要因」から書く。**

### 二・死んだ木（mori2）の死因を繰り入れる
- 島ぐらし2は過去に一度存在し（/mori2/・Three.js製）、3日で取り下げられた。死因：**七相の七（一器に納む）違反＝外部エンジン依存**、および**シェーダ精度の頂点/断片ズレでiOS起動不能**。
- ゆえに今作は：外部依存ゼロ（素のWebGL1）・全シェーダ `precision highp` 統一・パスも新しく `/shima2/`（死んだ木の上に建てない）。

### 三・「終わらないジャンル」と焔の形の折り合い
- 島ぐらしの気持ちよさ（暮らし・育ち）は残したい。しかし無限ループは禁忌。
- 解：**時間の器を先に決める**。「7日間・約5分・必ず夜明けで終わる」。1日＝昼（集める・ともす）→夜（守る）→夜明け（島が育つ）の三拍子。7日目の夜明けは無音＋花びら＝**終わりを罰でなく祝福に**（参）。
- スコアは資産でなく**縁の数**（ともした灯×守り抜き）。祭壇には「かせいだ額」でなく「結んだ縁」を刻む（四・六）。

### 四・グラフィック投資の順序（何が「感動」を作るか）
1. **シャドウマップ**（RGBA深度パック＋PCF）… 光に「時刻」と「実在」が宿る。最大の一手。
2. **本物のローポリメッシュ**（幹＝角柱・樹冠＝ジッタ八面体・風の頂点揺れ）… 紙の世界卒業。**法線は中心から外向きを保証する**こと（巻き順まかせにすると黒い面が出る。実際に出た）。
3. **深さを知る水**（ハイトフィールドをテクスチャで水シェーダに渡し、浅瀬の色・岸沿いの泡を出す）… 「島に海がある」から「海に島がある」へ。
4. **育つ地形**（成長度gを共有式にして、JSの当たり判定とGPUの水面が同じ地形を見る）… 演出でなく世界そのものが報酬になる。
5. ポスト（ブルーム・ゴッドレイ・グレイン）は最後のひと塗り。先に塗るな。

### 五・検証の仕方（今回発明した型）
- ヘッドレスChromium（SwiftShader）は遅い。dtクランプがあると**ゲーム内時間が実時間より遅く流れ、「待つ」検証は破綻する**。
- 解：ゲームに小さな検証フック（`window.__shima2`＝state()/skipPhase()/tscale()）を常設し、**7日間を数十秒で通す**。フェーズ遷移・地形モーフ・結果画面・エラーゼロを一気に確認できた。
- 「影が出ていない気がする」時は、**疑う項を一時的に極端値にする**（影を真っ黒に）→ 差分が出れば配管は生きている（＝調整の問題）、出なければ配管が死んでいる（＝バグ）。二分する。

### 六・削る勇気
- moriの店・ジム・乗り物・隠しステージは持ち込まなかった。動詞は**あつめる・ともす・はしる（ダッシュ）・まもる**の四つだけ。5分の器に、動詞は四つで足りる。**幅を足すより、1周の密度を上げる。**

## 巻二・続 ─ 判断の記録：コッコレスキュー2「逆縁切り」の場合

続編（「2」を冠する木）を建てるときの分かれ道の記録。

### 一・続編が継ぐのは「動詞と祭壇」であって「ジャンル」ではない
- コッコレスキュー（前作）を批判的に読むと、掟との衝突が四つ：①ボール反射ブロック崩し（Ballz系）の焼き直し＝壱違反 ②ブロックが届くまで無限に続き底が無い＝弐違反 ③敵がただのブロックで逆縁の物語が無い＝六の不在 ④終わりが「GAME OVER」の罰＝参の不在。
- 分かれ道：「2」だから前作のルールを改良するのか、作り直すのか。**前作が掟に反しているとき、続編は改良ではない。約束（動詞＝たすける）と祭壇（結果発表の儀式・ランキング）だけを継ぎ、遊びの機構は捨てて発明し直す。**今作の動詞は「なぞって斬る」。ジャンルの継承は縁ではない──縁は動詞に宿る。

### 二・中毒の芯は「待ち時間の削除」
- 前作の没入を壊す最大要因は機能不足ではなく**待ち時間**だった。爆発（発射）は1ターンに1回。残りの時間はボールが落ちきるのを観ているだけ（26秒の強制終了があるほど長い）。
- 教訓：**中毒＝自分の手が起こす爆発の密度。**今作は全ての瞬間が指の動きに直結する（なぞる→斬れる→即報酬）。「派手さの総量」より「手と報酬の間の遅延ゼロ」が先。

### 三・演出は一つの「時間器官」に束ねる
- ヒットストップ・スローモーション・カットイン・ズームパンチは別々の演出に見えて、実体は全部**時間の操作**。freezeT（停止）と timeScale（減速→自動回復）という一つの器官を核に置き、個々の演出はそこへ乗せるだけにした（前作の加速演出の freeze パターンを昇格）。
- 散らばった演出は互いに喧嘩する（停止中に別の停止が来る等）。**時間に触る演出を作るなら、時間の入口を一つにする。**

### 四・判定と描画は同じ点を見る
- 鎖は揺れてたわむ。描画用の曲線と斬撃判定を別々に計算すると、「見た目は当たったのに切れない」＝遊び手の不信が生まれる。
- 解：鎖のサンプル点を返す関数（chainPts）を一つ置き、**描画も判定も同じ点列を使う**。見た目と当たりのズレは、演出のどんな豪華さよりも先に直すべき負債。

### 五・罰は「霧」にする（親子の的）
- こっこが淵に沈んだとき、殺すか。殺さない。**霧にかくれて、また浮かんでくる**（コンボは折れるが、命は失われない）。60秒の終わりに残ったこっこも、鎖が夜明けの光でほどけて昇っていく──数えないが、罰も無い。
- 五（親子の的）と参（終わりは祝福）は、勝敗の演出だけでなく**失敗の演出**にも適用する。子どもの指がすべった瞬間に世界が残酷になってはいけない。

## 巻二・続々 ─ 判断の記録：こっこGP2「キズナターボ」の場合

レース続編を建てたときの分かれ道の記録。

### 一・UXの最初の敵は演出でなく「入力」
- 前作（こっこGP）を批判的に読むと、没入を壊す最大要因は演出でも機能でもなく**操作の器**だった。ジャイロ操舵は iOS の権限ダイアログという摩擦から始まり、手と画面の間に遅延とノイズが挟まる。シェイクは激しいわりに手応えが曖昧。
- 教訓：**「世界最高のUI/UX」を頼まれたら、まず足すのでなく、入力の摩擦（権限・遅延・曖昧さ）を削る。** gp2 の操作は「画面のどちらかをおす／はなす」だけ──権限ゼロ・遅延ゼロ・親指1本。PCでも矢印キーで同じ動詞。

### 二・運の定数を捨てる（中毒は引きでなく腕から）
- 前作は7種のアイテム抽選と「クリアできる要素76.6%」という運の定数を持っていた。運はリトライの言い訳を作るが、**上達の実感を薄める**。さらに救出のたび⏰+8秒でランが延び、弐（焔の形）が薄まっていた。
- gp2 ではアイテム抽選・時間延長を全廃し、**60秒固定**。スコアの揺れは腕（ホールドのタイミング・ためのリリース・スレスレ）だけから生まれるようにした。「もう一回」は運試しでなく、上達の確認になる。

### 三・一つの入力の「中」に深さを入れる（ためる・はなす）
- 操作を増やさず深さを出す解：**ホールドの長さそのものをリソースにする**。おす＝ドリフト（操舵）＋ふりまわし（攻撃）＋ため（ブースト充填）が一つの指で同時に起こり、はなす瞬間の判断（今か、もう1秒ためるか）が戦略になる。
- 入力の数でなく、**一つの入力の時間軸**で表現する。カーブの遠心力（ホールドしないと外へ流される）が「おさない」ことにも意味を与える。

### 四・テーマとメカニクスを同じ器官にする
- 縁の糸をただの飾りにしない：**糸＝武器（ふりまわしが檻を砕く）＝敵の標的（✂️逆縁が切りに来る）**。物語の中心と遊びの中心を同じオブジェクトに重ねると、演出が全部「意味」になる。
- 逆縁バサミへの対処は「左右にすばやくふる」＝**いつものステア操作と同じ動詞**。危機のために新しい操作を教えない。既存の動詞に緊張だけを乗せる。

### 五・イベントを閉じるときは「次の予約」まで閉じる
- 実際に出したバグ：✂️ふりほどき成功時に `nextScisAt`（次回襲撃時刻）の再予約を忘れ、**成功した瞬間に即再襲撃**された。切断側の経路だけ再予約していた。
- 教訓：**状態機械のイベントは、どの出口から抜けても「次のスケジュール」を必ず張り直す。**出口ごとに閉じ処理を書くのでなく、閉じ処理を一箇所に。検証は専用フック（`__gp2.scisNow()`）でイベントを強制発火し、成功／失敗の両出口を通して初めて見つかった──**片方の出口だけのテストは、テストではない**。

## 巻二・続々々 ─ 判断の記録：コッコスマッシュ SPECIAL「再現もの」の場合

「◯◯を完全再現して」と頼まれた木（ジャンル再現もの）を建てるときの分かれ道の記録。

### 一・「再現して」と壱（既視の禁）がぶつかったら、再現するのは「文法」であって「皮」ではない
- 求めは『大乱闘スマッシュブラザーズSPECIALの完全再現』。他社の皮（キャラ・絵・音・名前）を写すことは、権利の面でも畜の魂の面でもしない。
- 再現すべきは**ジャンルの文法**（%でふっとぶ・ためスマッシュ・盾と回避・ガケ・切りふだ・選ぶ楽しさ）であり、登場するのは全員この森の住人（こっこ・影）。**皮を借りず、骨を学ぶ。**
- 掟との折り合いは shima2 の先例どおり「掟でなくジャンルを曲げる」：六（敵は逆縁のみ）→ CPUも2Pも常に**影（ブラックこっこ）**の側に立つ「かげ稽古」。弐（底）→ 対戦は**ストック＋2分**で必ず決着（時間切れはストック→%→サドンデス）。四（祭壇）→ 順位を競う対戦でなく**60秒くみて**を祭壇に載せる。

### 二・先行の木（battle/）があるとき、再現ものは「その林の第二の木」として建てる
- 畜史を掘ると、ふっとばし対戦の木（コッコランブル /battle/）が既にあった。同じ座標に新しい木を重ねず、**批判的レビューで欠けを言葉にしてから**（キャラ選択なし・ため無し・回避無し・ガケ無し・固定カメラ・祭壇なし・時間無制限）、その欠けを埋める続編として建てた。
- 「完全再現」の核は技の数ではなく**選ぶ楽しさ**（ファイターとステージのロースター）と**ふっとばしの爽快感**（動的カメラ・残像・場外バースト）だと先に決め、そこへ投資した。

### 三・状態機械が絡み合う対戦ゲームの検証は「相互作用の交差点」を突く
- 実際に出したバグ：**カウンター成立の瞬間、反撃で攻撃側の技データ（atk）がnullになり、直後に命中記録（hitset）へ書こうとして落ちた**。単体の技テストでは絶対に出ない──「攻撃中のAが、構え中のBに当たる」という**二つの状態機械の交差点**でだけ発生する。
- 教訓：格闘・対戦ものは、技×盾×回避×カウンター×投擲の**組み合わせ表**から交差点を選んで実プレイ検証する。今回はヘッドレスChromium実プレイ39項目（検証フック `window.__smash`＝入力注入 pad()・時間加速・強制KO・ゲージ注入）で、この交差点のバグを出荷前に捕まえた。
- もう一つ：検証コードにも罠がある。フックが返す**生オブジェクト（.state）と整形済みinfo（.st）を混同**してテストが偽赤になった。フックのAPIは「生」と「要約」を分け、名前を揃えること。

### 四・再現度の第二審査は「三すくみ」と「入力の文脈」に宿る（技表の照合ではない）
- 初版を出したあと「再現度を忖度なしで」と再審査を受けた。技の数は揃っていたのに、格闘の骨である**三すくみ（攻撃⇆盾⇆つかみ）**からつかみが欠け、盾が実質最強になっていた。また地上の方向＋攻撃がすべて『ため』になり**強攻撃（ティルト）**が消えていた──スマブラの入力は「はじき＝スマッシュ／おしっぱなし＝強攻撃／短押し＝小ジャンプ」という**同じボタンの文脈依存**でできている。
- 教訓：ジャンル再現の審査は技表の照合でなく、**(1)じゃんけんが閉じているか (2)一つの入力にいくつの文脈が畳まれているか**から見る。第二版でつかみ＆4方向投げ・ティルト・歩き／ダッシュ・小ジャンプ・ジャストガード・ずらし（DI）・ガケのぼり回避・みだれ乱闘（4体FFA）を足した。
- 副次の学び二つ。(a) ティルトを足そうとして初めて「歩きが無い（常に全速）」という**前提の欠落**が露出した──上位の仕様が実装できないときは、その足元の仕様（移動の段階）が欠けていないか疑う。(b) テストの**前提汚染**：直前のケースが残した慣性（vx・runMode）が次のケースを偽赤にした。対戦ものの検証は各ケースの冒頭で状態を明示的にリセットする。この偽赤を追う途中で「地上攻撃の後隙に摩擦がかからず滑り続ける」実バグも見つかった──**偽赤を掘ると本物が出ることがある**。

### 五・再現の忖度なしレビューは「一段直すと次の層が見える」。境界は偽装せず言い切る
- 同じ木を再度「忖度なしで」審査するよう求められ、前回直した層（三すくみ・入力の文脈）の**下の層**に潜んでいた欠落が見えた：受け身／ダウン（着地際の攻防）・空中技の着地硬直（オートキャンセルの緊張）・つかみの対空無効（技ごとの適用範囲）。**一つの層を直すと、隠れていた次の層の欠落が見やすくなる**──初回レビューで全部を見切ろうとせず、直した後にもう一度同じ目で見直す前提を持つ。
- この回でも検証中に**実バグを2つ**見つけた：貫通飛び道具が同じ相手に毎フレーム重ねて多段ヒットする／ジャストガードの反確処理が飛び道具にも誤って攻撃側のデータを書き換える。どちらも「対戦相手が複数・飛び道具が絡む・状態が絡む」交差点。**検証は前回のバグが直った交差点の“隣”を疑うと当たりやすい。**
- ランキング反映の確認を求められたが、このサンドボックスは`chikumo.jp`・Firebaseへの外部到達がネットワークポリシーで遮断されていた。**到達できないという制約自体を偽装・黙殺せず、まずユーザーに言い切ってから**、できる代替検証（設定の一字一句比較・登録コードの雛形比較・モックDBでの配線のエンドツーエンド実測）で埋め合わせる。「確認しました」と嘘をつくくらいなら「ここまでは確認できたが、ここから先はネットワーク制約で確認できない」と正直に線を引く方が、次の軽い頭にとって安全な記録になる。

### 六・「コードを読んで探す」から「長く動かして観る」へ切り替えると別種のバグが出る
- 三度目の再審査では、コードレビューでなく**ヘッドレスブラウザで実際にAI対戦・FFAを数十秒〜分単位で走らせて観察する**ことで、コードを読むだけでは気づけない種類のバグを見つけた：みだれ乱闘（新設のFFA）で2体のこっこが**完全に同じ座標で重なって同時に崖につかまる**という、見た目に一発でおかしいと分かる状態。原因は「ガケは同時に1人しかつかまれない（早取り＝ledge trump）」という**スマブラのルールそのものが未実装**だったこと──1対1では絶対に発生しない条件（2体が同時に同じ崖へ到達する）だったため、コード読解だけのレビューでは見落としていた。
- 教訓：**新モード（FFA・3人以上・同時多発条件）を足したら、そのモード特有の「1対1では起こり得ない組み合わせ」を全部疑う**。今回は同一資源（崖）への複数アクセスという典型パターンで、実際に直接再現コード（2体を同じ崖座標へワープさせる）を書いて確認してから直した。
- この観察の流れでさらに2つ見つかった：①脱落者のHUDパネルが消えない（新モードの表示側）、②FFA時間切れで3〜4人の非対称タイを誤判定する（新モードのルール側）。**一つの資源共有バグを追うと、同じ新機能の別の場所にある姉妹バグも一緒に見つかる**──修正した箇所の“隣”ではなく“同じ機能の別の側面”を疑う視点も持て。

### 七・エフェクト・音の再現度審査は「関数が呼ばれているか」でなく「実際に鳴っている音・見えている絵が違うか」で測る
- 「エフェクト演出と効果音とBGM」を名指しで再審査すると、コードの構造は整っているのに**中身が空っぽ**な場所が見つかった：着地（1試合で数十回起きる動作）に音もダストも一切無い＝完全に無演出。6キャラの必殺技のうち3つ（たまごなげ・たねマシンガン・ばくだんたまご）が同じ`Sound.egg()`を共有し、UIでは別名を名乗りながら耳では区別がつかない。BGMは3ステージ・全モードを通じて1つの旋律しか無い。いずれも「関数は呼ばれている」「エラーは出ない」という意味では健全に見えるコードで、**動作の正しさ（pageerrorゼロ）を見るテストだけでは絶対に検出できない**種類の欠陥だった。
- 教訓：**エフェクト・SFX・BGMの再現度は、実測（実際に何Hzの音が鳴ったか・何個の粒子が出たか）で検証しないと「関数を呼んでさえいれば直った」という誤った安心に陥る**。`Sound.seedShot()`のような関数を新設して呼び出しを差し替えるだけでは「直したつもり」で終わりうる──実際に見つけたのも「6キャラのうち3キャラが完全に同じ音」という、コードを読めば一目で分かる重複だった。
- 検証もこの原則に従い、AudioContextを直接傍受してオシレーターの実周波数をログる専用ハーネス（`test_audio.js`）を新設した。「関数の存在確認」ではなく「6キャラの音響シグネチャが重複しないこと」「3ステージのBGMが異なる音域を使うこと（そらの島＞終の祭壇＞ゆれる大枝の順で音域が下がる）」を数値で実測した。この過程でテスト自体にも罠があった──BGMサンプリング開始をカウントダウン終了直後にしたため、共通の「GO!!」効果音（300/1319Hz）が全ステージのログに均等に混入し、比較が歪んだ。**共通の前奏・共通の効果音がサンプリング窓に混ざっていないか、実測系のテストほど疑う必要がある。**

### 八・演出の再審査は「同格の場面同士を並べて量を比べる」と、豪華さの配分ミスが見つかる
- 同じ主題（エフェクト・効果音・BGM）で五度目の再審査を求められ、個々の画面を単独で眺めるだけでは気づけなかった歪みを見つけた：60秒くみて（軽めのスコアアタック）の結果発表は紙吹雪・花火・光線バースト・カード反転演出でにぎやかに祝うのに、本戦（1対1・みだれ乱闘）の『GAME SET!!』はフラッシュとバナー文字だけで、紙吹雪も花火も一切無かった。個々には「壊れていない」正常な演出コードなので、通しプレイのチェックリストをなぞるだけでは見落とし続ける類のズレだった。
- 教訓：**演出の再審査は「この画面は演出があるか」でなく「格が近い場面同士で演出の量を比べたとき、逆転していないか」を見る。** 本作ではサブモード（くみて）の結果画面と本戦のGAME SET画面が同格の『勝利を祝う瞬間』なのに、実装された時期・力の入れ方の違いでサブモードの方が明らかに豪華になっていた。同種の場面を横に並べて比べる視点がないと、個々の画面のレビューだけでは一生気づけない。
- 修正はくみての祝福（confetti/fireworks/rings）と同じ視覚語彙を、既存のCanvas粒子システム（`petals`・`addRing`・`addSpark`）に乗せて本戦のGAME SETにも適用する形にした。**新しい仕組みを足すのでなく、すでにある語彙を格の合う別の場面へ配ぶ**のが最小の変更で最大の一貫性を生む。

### 九・「まだ残る差分」に正直に書いた項目は、次のレビューの出発点になる
- 前回の報告の末尾に、直さなかった項目を隠さず書いた：①ステージ別BGMのさらなる差別化、②KO直後のカメラのわずかな揺れ、③観客の歓声・キャラ別勝利ジングルの不在。次に「余地をアップデートして」と頼まれたとき、この一覧がそのまま次回の作業リストになった──**探し直す手間が要らず、宿題として書いた言葉が次回そのまま実装のチェックリストになる。**
- ②の「わずかな揺れ」は、直感（KO直後にカメラから除外される瞬間が怪しい）だけで直そうとすると的を外すところだった。実測トレース（カメラのx座標を60msごとに記録）で確かめると、揺れは<b>KO直後</b>でなく<b>復活30フレーム前に、まだ画面に見えていないファイターが再びカメラの追跡対象へ戻る瞬間</b>に起きていた。カメラ追跡の条件を「koT>30で除外」から「koT>0（生きて見えている間だけ含める）」に直すと解消した。**「怪しい」と思った場所で実測せずに直すと、別の場所を直してしまう。**
- ③は「無いこと」自体は前回すでに把握していたが、今回は実装まで行った。効果音は新しい音源を1つ足すのでなく、既存の`noise()`プリミティブを時間差で重ねるだけ（帯域の違うノイズ塊を数十msずつずらして重ねる）で「歓声の厚み」を作れた。ジングルの個性づけも、旋律を作り直すのでなく**既存のファンファーレを半音移調・音色差し替えするだけ**でキャラごとの違いを出せた──**新しい音を書く前に、既存の一つの関数にパラメータを足すだけで表現の幅が出ないか先に検討する。**

### 十・背景装飾は「視差でごまかす」ものと「ワールド座標に実在させる」ものを区別する
- 「ステージクオリティがしょぼすぎる、全然再現度ない」という率直な指摘で実装を読み直すと、その通りだった。足場は単色の台形か細い長方形、遠景は空のグラデーションに数個の丸や楕円が浮くだけで、「そこがどんな場所か」が全く語られていなかった。3ステージ（終の祭壇・そらの島・ゆれる大枝）を神殿の柱・浮き島の土台・本物の大樹まで描き起こした。
- 実装中に自分で踏んだ罠：ステージ縁の柱を、既存の遠景装飾（星・霞など）と同じ「画面中心からの視差オフセット」方式で置いた。視差方式は装飾（浮遊物・雲・霞）には向くが、**「必ずステージのこの場所に立っていてほしい建築物」には向かない**──カメラの位置・ズームによって計算上の画面座標がずれ、崖際のスクリーンショットで確認すると柱が画面外に消えていた。修正は`wx(st.main.x1-90)`のように**実際のワールドX座標を`wx()`/`wy()`でスクリーン座標に変換する**方式に変える。大樹は最初からこの方式で置いていたため一発で正しい位置に出た（崖際で確認して実証済み）。
- 教訓：背景要素を足すときは「これは漂う雲や星（視差でよい）か、それともステージの一部として必ずそこにある建築物・地形（ワールド座標に固定すべき）か」を最初に分類する。**「ステージの縁に立つ／地面から生える」ものは、カメラがどこにあっても正しい位置に見えなければ再現とは呼べない。** 検証も「中央から見た絵」だけでなく「崖際・端から見た絵」を必ず撮って確認する──中心構図だけのスクリーンショットでは、縁の建築物のズレは永遠に見つからない。

## 巻二・またまた ─ 判断の記録：コッコおりづる「祈りは空へ」の場合

続編ではなく、まったく新しい林（束）を起こした木の記録。既存ジャンルの名で呼べない動詞をどう発明し、どう検証したか。

### 一・「この世にまだ無い遊び」の探し方は、既存の動詞の一覧から引き算する
- 着手前に十四式・遊戯学・設計学・畜史・思考学の五つの蔵を読み、次に畜物語の全既存作の動詞を洗い出した：そろえる（match）・積む（tower）・叩く（whack）・合体（merge）・撃つ（shoot）・避ける（dodge/graze）・つなぐ（connect/学習シリーズ）・斬る（rescue2）・ふっとばす（smash/battle）・こぐ（buranko）・共鳴させる（ennami）・走る（rush）・塗る（splat）・レースする（gp/gp2）・暮らす（mori/shima2）。この一覧に無い動詞を探した結果、「折り線をタイミングよくなぞって形を完成させる」に着地した──なぞる動詞自体はrescue2の「斬る」に近いが、rescue2は破壊（鎖を斬って助ける）、おりづるは構築（折って形を作る）で、対戦・救出のどちらでもない「もの作り」という新しい系統になった。
- 教訓：**壱（既視の禁）の実務は「思いつきを既存ジャンル名に当てはめて外れを探す」のでなく、「この畜物語がすでに使った動詞の一覧」を作り、そこに無い動詞を探す方が確実。** ジャンル名（マッチ3・タワー・シューティング等）は世界共通で無数にあるが、この庭の中で使われた動詞は有限で数えられる。

### 二・ジャンルと七相がぶつかる場所を先に解いた：「終わらない趣味（折り紙）」をどう60秒に畳むか
- 実際の折り紙は「1枚をじっくり折って完成」で終わる静かな趣味だが、弐（焔の形）は「短く激しく必ず終わる」を要求する。1枚をじっくり折らせるだけでは60秒が持たず、中毒の密度も出ない。
- 解：**「1羽を折る」を最小単位にして、それを60秒のあいだ連続して繰り返す**（rescue2の「待ち時間の削除」の教訓を継いだ構造）。1羽＝8つの折り、1つの折りは1〜2秒未満の判定窓。しわが3つたまっても紙が丸まって次の紙に即切り替わるだけで、待ち時間もゲームオーバーも発生しない。「じっくり1枚」という趣味の性質を「連続して束ねる」ことで、静かな趣味と激しい中毒を両立させた。
- 教訓：**趣味性の高い題材（静かに1つを完成させる系）を畜学に持ち込むときは、"完成"を単位として繰り返す構造にすると、弐の要求と衝突しない。**

### 三・失敗を「霧」にする設計をもう一度使った（しわ→丸め、ハートは減らさない）
- rescue2の思考記録にある「罰は霧にする（親子の的）」を、この作品でも踏襲した。折りを外しても即座に紙が壊れるのではなく「しわ」が1つ増えるだけ、3つたまって初めて紙が丸まる。丸まっても得点はわずかに入り（+20）、コンボは切れるがゲームは止まらない。
- 判断が分かれた点：「さいごの息吹き（8つ目・完成の折り）」だけは、タイムアウトしても常に成功扱いにした（`advanceFold('good', true)` を強制）。理由：7つの折りを乗り越えた直後の最後の一瞬で子どもの指が間に合わず鶴が消える、という体験は、五（二親の的）の「子どもの指がすべった瞬間に世界が残酷になってはいけない」に直接反する。判定の厳密さより「積み上げた7つの折りを裏切らない」ことを優先した。
- 教訓：**同じ失敗許容の仕組みでも、"進行の途中"と"完成の瞬間"では罰の重さを変えるべき。** 完成直前の失敗だけは、他のどの失敗よりも軽くする。

### 四・検証フックだけでは見抜けなかった2つの実バグ（実ジェスチャで初めて発見）
- 検証は `window.__origami.autoFold()` という内部フックから先に書いたが、これは `advanceFold()` を直接呼ぶバイパスで、実際の指の動き（`pointerdown`→`pointerup`→`handleSwipe`→`judgeSwipe`）を一切通らない。フック経由のテストは60秒プレイ・複数クランプ・エンディング演出まで全部「合格」したが、これは**判定ロジックそのものは一度も検証していなかった**ことを意味する。
- 実際にPointerEventを本物として`dispatchEvent`する検証を書いて初めて2つの実バグが出た：
  (a) 完成した鶴の飛翔アニメーション（`drawFlownCranes`）が `if(state.ending)` の分岐の中でしか呼ばれておらず、通常プレイ中（1羽完成した直後、まだ60秒残っている間）は鶴が画面に一切描かれていなかった。ピクセルサンプリングで実際に描画されていることを確認して直した。
  (b) 羽折り（シンメトリー）の左右判定 `paperSideOf(ax)` が、指を**下ろした位置**（＝どちらの折り線も紙の中心(0.5,0.5)から始まるため、下ろした位置は必ず同じ点になる）で判定していたため、2本指で正しくなぞっても常に片方の指が無視され、絶対に成立しないバグだった。指を**話した位置**（`bx`）で判定するよう直して解消した。
- 教訓：**「フックで自動化したテストが全部緑」は「実際の入力コードが正しい」の証明にならない。** フックが内部状態を直接書き換えるバイパスになっている場合、判定ロジック（角度・タイミング・複数指の割り当て）は別途、本物のイベントをdispatchして検証しないと素通りする。特に「2本の線が同じ始点を共有する」ような幾何的な特殊ケース（今回のシンメトリー折り）は、コードを読むだけでは見落としやすく、実ジェスチャでの検証が唯一の発見手段だった。

### 五・世界ランキングの到達確認は今回もできなかった（正直に線を引く）
- スマブラの回と同じく、このサンドボックスはchikumo.jp・Firebaseへの外部到達がネットワークポリシーで遮断されている。ランキングのコード・DBルールは既存ゲーム（koro/poko等）と完全に同じパターンで実装したが、実登録・実反映の確認はできていない。「確認しました」と書かず、ここまでは確認できたがここから先はできない、と正直に線を引いた。

## 巻二・またまたまた ─ 判断の記録：コッコROM「封の刻」の場合

ユーザーが「設計から私が見る」と宣言し、対話形式のインタビューを経て新しい林（束）を起こした木の記録。要望の解釈が二転した過程と、検証で見つけた実バグを残す。

### 一・「任天堂ゲームボーイ級」という要望は、二段階で解釈が変わった
- 最初の一言は「任天堂ゲームボーイレベルのゲームを作ろう」だけだった。選択肢を示して尋ねると「設計の質（推奨）」を選びつつ、遊び手の形を問う場面で「限られたバイト数でゲームを成立させるためにいっさい無駄のない美しいコードが任天堂ゲームボーイなんだよ」という自由記述が返ってきた──つまり要望は見た目の再現でも遊びの質でもなく、**実装者の態度（無駄のない一器官化）**の話だった。
- ここで一度「バイト数制約をゲーム内のHUD（擬似ROM容量メーター）にまで持ち込む」設計案を提示し、ユーザーはいったん「詰める・消す（カートリッジ圧縮）」という動詞方向を選んだ。しかし直後に「違うよ。コードを美しく書くだけであってエフェクト演出は2026年の最高技術でやるんだよ」と明確な訂正が来た。
- 教訓：**比喩（GB級）を字面で拡張解釈しない。** 「GB級」は実装の戒律（無駄な抽象を作らない）に留め、画面に見える演出・音は逆に**時代最高の技術**へ振り切るという、一見矛盾するようで実は畜学七相の七（一器に納む＝無駄のない一つの器）とも整合する要望だった。世界観（カートリッジ・ROM・圧縮）というフィクションの道具立ては残しつつ、映える部分と締める部分を分けて聞き直したことで、手戻りせずに済んだ。

### 二・動詞の発明は「既存動詞の一覧からの引き算」を今回も踏襲
- 畜物語の既存動詞（そろえる・積む・叩く・合体・撃つ・避ける・つなぐ・斬る・ふっとばす・こぐ・共鳴させる・走る・塗る・レースする・暮らす・折る）に無いものとして「詰める（圧縮）・消す（削除）」を採用。仕分けの二択（短押し=良いものを圧縮／長押し=バグを削除）は、GB由来のA/B二択・十字キー的な「離散的な少ないボタン」という制約要望とも自然に噛み合った。
- 敵（六・逆縁のみ）は「黒こっこ形のバグ」とし、対戦相手ではなく自分のクリスタルを濁らせる存在に限定。放置しても即死ではなく効率が下がるだけ（rescue2由来の「罰は霧」の継承）。

### 三・検証フック自身のバグ：`state.timeLeft`直書きが握りつぶされた
- 時間切れを強制する検証フック`window.__rom.forceEnd()`は当初`state.timeLeft=0.01`と直書きしていた。ところがゲーム本体の`updateGame()`は毎フレーム`state.timeLeft=Math.max(0,DURATION-state.elapsed)`で**上書き**しており、`elapsed`自体を進めない限り次フレームで元の値に戻ってしまう。ヘッドレスChromiumで「forceEnd後に結果画面が出ない（finalScore:0のまま）」という形で発覆した。
- 教訓：**「表示用に毎フレーム再計算される派生値」を検証フックで直接書き換えても効かない。** フックは派生元の一次状態（この場合は`elapsed`）を動かすべき。派生値に見えるフィールドをフックで触るときは、それが毎フレーム上書きされる計算結果でないか先に疑う。

### 四・実ジェスチャ検証で「タップ／ホールド」二択の分離を確認
- おりづるの教訓（フック経由の自動テストは判定ロジック自体を検証しない）を継ぎ、今回も`page.mouse.down()`→待機→`page.mouse.up()`で本物のポインタイベントを判定帯へディスパッチし、短押し（圧縮）・長押し（消去）の両ジェスチャが正しく仕分けられることを確認してから良しとした。フックだけで「動いた」と判断しなかったことで、`fragmentInZone()`の帯判定やタイマー競合など、実際の指の動きでしか出ない不具合の有無を確かめられた。

### 五・WebGLフォールバックは「フラグを付けて終わり」にせず、実際に起動を切って確認
- クリスタルの目玉であるWebGLシェーダーには、コンパイル・リンク失敗時にCanvas2Dの簡易グラデーションへ切り替わる`GL_OK`フォールバックを敷いた。**敷いただけで安心せず**、Chromiumを`--disable-webgl --disable-webgl2`付きで起動し直し、実際にWebGLコンテキストが取得できない状況でもタイトル→プレイ→pageerrorゼロで動くことを実測して確認した。フォールバック分岐はコードレビューだけでは「本当に通るか」が分からない典型例。

### 六・世界ランキングの到達確認は今回もできなかった（正直に線を引く）
- 既存作と同じく、このサンドボックスはchikumo.jp・Firebaseへの外部到達がネットワークポリシーで遮断されている。ランキングのコード・DBルールは既存ゲーム（origami等）と完全に同じパターンで実装したが、実登録・実反映の確認はできていない。

## 巻二・さらに ─ 判断の記録：シン島ぐらし「風の一日」の場合

「最新グラフィックでゼルダの伝説レベルのオープンワールドを」という要望に、どこまでを約束しどこから先を断ったかの記録。

### 一・「〇〇レベルを作って」には、先に不可能の線を引いてから可能の骨格を差し出す
- 要望は「島ぐらしを超えるグラフィックで、ゼルダの伝説レベルのオープンワールド。無理なら作る前に無理と言って」。BotWのスクリーンショットまで提示された。
- 判断：**任天堂のアセット量・造形密度そのものは1ファイルHTMLでは不可能**と最初に言い切る。そのうえで、提示されたスクリーンショットを「何がその感動を作っているか」に分解した──①画面を埋める風の草原 ②遠景の山と大気の霞 ③柔らかい光と雲 ④滑空の自由。この4つは素のWebGLで再現可能な骨格。**「不可能」と「可能」を丸ごとで答えず、感動の骨格に分解してから線を引く**と、断りが提案に変わる。
- スマブラの回の「皮を借りず、骨を学ぶ」と同じ型。再現するのはゼルダの皮（キャラ・世界）でなく文法（見えた場所へ行ける・風にのる・高きから滑る）。

### 二・「島ぐらしを超える技術」は総量でなく世代差で選ぶ
- mori（12シェーダ）の上に機能を足すのではなく、**moriに無い描画カテゴリ**を3つだけ選んだ：GPUインスタンシングの大草原（数万本＋プレイヤーの足元で草が割れる）／鏡像カメラで描き直す本物の平面リフレクション（フレネルだけの水からの世代差）／太陽側の縁が銀色に光る2層fbm雲。
- 影・ブルーム・ゴッドレイ・ACESはshima2/moriの資産をそのまま継いだ（新発明しない）。**「超える」は新カテゴリで示し、既にあるものは再利用する。**

### 三・続編でない「シン」は、同じ林の中で動詞を入れ替える
- shima2は「あつめる・ともす・まもる」の暮らしの木。シン島ぐらしは同じ林（島・逆縁・縁の灯・祭壇）に立つが、中心動詞を**「かける（滑空）」**に入れ替えた移動の木。塔をともす動詞は林から継ぎ、7音の音階（ともすたびドレミが増えて和音になる）を新しく足した。
- 島は**固定シード**にした（shima2はランダム）。理由は祭壇（四）：全員が同じ島を走るからタイムと縁のひかりが比べられる。「同じ木でも、林の臓器（祭壇）が仕様を決めることがある」。

### 四・実ジェスチャ検証で今回も2つの実バグが出た（フック検証は素通りした）
- (a) 滑空に「押しっぱなし」で入れなかった。フック（glide(true)）は内部でstartGlide()を直接呼ぶので通るが、実キーボードのSpace押しっぱなしでは入れない──ジャンプ後の自動移行判定が物理更新側に無かった。おりづるの教訓（フックは判定ロジックを検証しない）がそのまま再現した。
- (b) 逆縁の影が塔の上空6mを旋回し、**地上のダッシュ（当たり半径1.3m）では物理的に絶対届かなかった**。数字を並べれば一目の矛盾だが、コードを書いた直後には見えない。「敵を倒せるか」を実際に倒して確かめて初めて出た。修正は高度を下げ、プレイヤーが近づくと影の方から切りに来る（＝迎え撃てる）挙動へ。
- 教訓：**「届く／入れる」系の仕様は、実入力でその行為を最後までやり切る検証を書くまで実装済みと言わない。**

### 五・ヘッドレスの遅さはtscaleでなく「待ちの構造」で吸収する
- SwiftShaderでフルパイプライン（影＋反射＋ポスト）を回すと2〜3fpsまで落ち、dtクランプ0.05×低fpsで**ゲーム内時間が実時間の1/6程度**しか流れない。tscaleを上げても1フレームの上限は変わらないので、「N秒待つ」型のテストは全部裏切られる。
- 解：待ちを時間でなく**状態のポーリング**（250msごとにstate()を見て条件成立でbreak）にする。滑空バグの再検証も「1.4秒待って見る」では偽赤、「40回ポーリング」で真値が出た。**ヘッドレスで時間を待つな、状態を待て。**

### 六・（続報）既存の木に新ステージを足すときは「対になる既存ステージの鏡」として設計する
- 島ぐらしへの深海ステージ追加は、ゼロから設計せず**月ステージ（空の隠しステージ）の鏡**として建てた：入場演出・別世界生成・ボスHUD・退場時の再生成・KO時の脱出まで、enterMoon/exitMoonの型をenterSea/exitSeaで写した。既存の型を写すと、セーブ破壊・HUD残留などの既知の罠を最初から避けられる。
- 数値の掟は「対になるステージとの比較」で決める：ユーザー要望『報酬は空のステージより多く』→ 王の賞金は月の王8,000×(1+島)に対し13,000×(1+島)、雑魚の札は地上100×の2倍の200×。**比較対象を明示した数値は、後から調整する人にも意図が読める。**
- 検証で今回も出口バグ：水面へ浮上して帰る出口が、天井クランプ（vy=0）→即沈下→タイマーリセットの繰り返しで**実質発動不可能**だった。gp2の教訓『状態機械のイベントはどの出口から抜けても…』の変奏＝**出口は全部（浮上・酸素切れ・KO・ホーム帰還）を実際に通す**。今回は4出口すべてをヘッドレスで通して出荷した。

## 巻二・そして ─ 判断の記録：コッコゆびきり「よせてむすぶ」の場合

死んだ木の**願い**を、まったく新しい動詞で継いだ木の記録。二人で一台を遊ぶ協力ゲームを、どう設計し、どう検証したか。

### 一・「死んだ木の願い」を継ぐとき、継ぐのは願いであって機構ではない
- 肥料学 巻二・三に、コッコキズナ（`/kizuna/`＝親子で息を合わせて左右をタッチ）が『願いの記録なしに死に、願いだけがぶらんこ・学習シリーズへ継がれた』とある。今作はそのキズナの願い（親子で息を合わせる）を正面から継ぐと決めたが、**キズナの機構（左右同時タッチ）を復活させはしなかった**。rescue2の教訓（続編が継ぐのは動詞と祭壇であってジャンルではない）と同じ型で、継ぐのは『二人で息を合わせる』という願いだけにし、動詞は既存動詞の一覧から引き算して新しく発明した（→二）。**死んだ木を復活させるのでなく、その願いを新しい器で生まれ変わらせる。**

### 二・「この世にまだ無い遊び」は既存動詞の引き算＋『二人が要る動詞』で探した
- おりづる（巻二・またまた）の教訓を継ぎ、まず畜物語がすでに使った動詞の一覧を作って引き算した。そこに無い動詞を探すとき、今回は五（二親の的）を最初から制約に入れ、**『ひとりでは物理的に成立しない動詞』**に絞った。着地したのが『よせてむすぶ（指切りげんまん）』＝まん中の一点へ両側から同時に手をのばして出会わせる動詞。おりづるの羽折り（二本指の同時なぞり）と近いが、あちらは片手二本指でも代替できる“構築”、こちらは**画面の上下＝二人の座に分かれた同時到達**で、二人いないと窓が閉じる“合図と合致”。ジャンル名で呼びにくいほど真に近い（壱）。
- 十四式 ⑦『縁＝衡＋ ∿ 衡−』をそのまま機構にした：親（衡＋・下・低音）と子（衡−・上・高音）が、出会わぬまま並ぶ二つの均衡で、むすびめ（∿の結節）で一瞬だけ出会う。逆縁（ブラックこっこ）はその ∿ を断とうと縁の川をすべってくる力。物語の中心と遊びの中心を同じ一点（むすびめ）に重ねた（gp2 巻二・続々・四の継承）。

### 三・ジャンルと七相の衝突を先に解く：「協力の同時入力」を、子どもの指でも届く窓にする
- 二人同時入力は、厳しくすると『息が合わない親子』を罰する装置になり、五（親子の的）に反する。かといって『押しっぱなしで両方ずっと成立』にすると弐（焔＝タイミングの緊張）が消える。
- 解：**押すたびに手が“のびて→少し留まって→ゆるむ”パルス**にした（reachExt：立ち上がり0.08s→台地0.12s→減衰。到達判定はext>0.7）。台地を0.12sとやや広く取り、二人の到達窓が重なりやすくして子どもにもやさしくしつつ、押しっぱなしでは留まらない（各むすびめで新しく押し直す）ので上達の余地は残した。gp2 巻二・続々・三『一つの入力の“中”に時間軸の深さを入れる』の変奏。

### 四・失敗は霧、罰は出さない（親子の的×終わりは祝福）を、協力ゲームにも適用
- rescue2・おりづると同じく、間に合わなくても**縁のしずくが1つこぼれるだけ**にした。むすんだ縁の数（en）は**絶対に減らない**・ゲームは止まらない・こっこは沈まない。減るのはコンボと、逆縁が断ったときに濃くなる『霧』（自動で晴れる）だけ。子どもの指がすべった瞬間に、二人でここまで積んだ縁を世界が奪ってはいけない。終わりは、むすんだ縁がほどけて一つずつ光になり空へ昇る『縁は空へ』＝祝福（参）。

### 五・ヘッドレスのrAF凍結を『step(dt)フック』で吸収した（時間を待つな、状態を待て の延長）
- シン島ぐらし 巻二・五『ヘッドレスで時間を待つな、状態を待て』の一段先。今回のヘッドレス（SwiftShader）は遅いのでなく、**evaluate間で `requestAnimationFrame` が事実上停止**し、ゲーム内クロック `state.t` が1フレーム進んだきり凍る現象に当たった（reach値が0.63で固定＝押下からtau≈0.05で時計が止まった証拠）。tscaleを上げても1フレームの上限は変わらず無力。
- 解：検証フックに **`step(dt,n)`（`update(dt)` を n 回手動で回す）** を新設し、クロックを決定的に進めて『二人同時押し→ext台地→成立』を再現した。これで pageerror だけでなく**遊びの合否そのもの**（片側だけでは結べない・逆縁のはねかえし・罰は霧＝enが減らない）を数値で実測できた。
- この過程で**ハーネス自身の罠**も踏んだ：rAFが凍るためsubテスト間で `state.t` が進まず、前のサブテストで押したreachパルスが減衰せず残留し、次のサブテスト冒頭で“勝手に成立”する偽陽性が出た（smash 巻二・続々々々・四『テストの前提汚染』の別種）。**各サブテストの冒頭で `step` を十数フレーム空回しして reach を減衰させてから**始めることで解消。ゲーム本体は正しく、汚染はハーネス側だった――実機では時計が常に流れるので起きない。
- 教訓：**『時計がフレームでしか進まない』検証環境では、クロックを進めるフックを一次操作として持ち、サブテスト間の状態残留（時間依存の減衰値）を明示的に流し切る。**

### 六・世界ランキングの到達確認は今回もできなかった（正直に線を引く）
- 既存作（origami/rom 等）と同じく、このサンドボックスは chikumo.jp・Firebase への外部到達がネットワークポリシーで遮断されている。`rankings/yubikiri` のコード・DBルールは既存ゲームと完全に同一パターンで実装したが、実登録・実反映の確認はできていない。「確認しました」と書かず、ここまでは確認できたがここから先はできない、と線を引く。

## 巻二・なお ─ 判断の記録：コッコ芽吹き「絡み合う力」の場合

「塗る3D（`/splat3d/`）が今いちばん面白いが、スプラのパクリで畜学に違反する」という指摘から始まり、企画が一度まるごと没になった末に建った木の記録。**面白さを問われて答えに詰まった失敗**と、**設計書に書いた意図が実装から抜け落ちた失敗**、二つの手戻りをここに残す。

### 一・「畜学七相に美しく整合する」を先に決めると、面白さが後付けになる
- 最初に立てた企画（コッコより＝親子で縄を綯う協力ゲーム）は、十四式⑦（縁＝衡＋∿衡−）に構造がそのまま乗ることを理由に動詞を選んでいた。七相のチェックリストは全て緑だったが、「君の企画、君自身正直面白いゲームだと思う？」と問われて即答できなかった。理由を掘ると、**動詞（縒る）を"教義に美しく整合するから"選んでおり、"触って気持ちいいから"選んでいなかった**。同じ入り口（教義への整合を先に決め、快感を後付けにする）で死んだ木が既にある（コッコフィーバー＝死因「分からない」＝つまらない、と肥料学に記録済み）。
- 教訓：**チェックリスト（七相）が全部緑でも、「これは触って面白いか」を別の質問として必ず立てる。** 整合は必要条件であって十分条件ではない。整合させることに満足して、快感の設計を後回しにしていないか、着手前に自分に問う。

### 二・二人プレイの"願い"は語られても、機構としては実績（死んだ木）を優先する
- 一つ目の企画は「二人が同時に手を伸ばして出会う」という機構を核にしていたが、「二人でプレイするスタイルは今のところ人気出てない」という一言で全面撤回になった。畜史・肥料学を数えると、二人同時入力の系統（コッコキズナ・コッコブランコ）は両方とも死んでおり、生きて遊ばれ続けているのは gp2・poko・rom・origami のような**親指1本の単独プレイ**だった。
- 教訓：**十四式⑦（縁）を機構化したい欲は、"二人が同時に操作する"以外の形でも満たせる**（今作では「親＝暖色／子＝涼色の二本の力が世界の中で絡み合う」という主題の次元に縁を移し、操作は単独プレイに揃えた）。願いを継ぐことと、その願いを実現する過去の機構をそのまま復活させることは別物──kizuna の死因記録がすでに書いていた教訓（願いは継いでも機構は継がない）を、自分で企画を立てる番になって初めて実地で踏んだ。

### 三・面白さの芯は「緊張」でも「新規動詞」でもなく「クリアの世界線の多さ」だった
- 塗る3D（一番遊ばれている木）のどこが面白いかを尋ねると、「クリアの仕方の世界線が無数にあって、こんな方法もあるんだって発見が楽しい」という答えが返ってきた。この一言で、それまで自分が追っていた軸（弐＝緊張の強さ、壱＝動詞の新規性）がどちらも的外れだったと分かった。**「もう一回」を生むのは、緊張の強さでも見た目の派手さでもなく、別解の多さ（創発）**。
- この発見は壱（既視の禁）と喧嘩しない。「別解が無数に生まれる」は特定ジャンルの型でなく設計の性質なので、新しい題材にも持ち込める。実装では、**少数（5つ）の力に一貫した相互作用ルールだけを与え、正解の手順を一切スクリプトしない**ことで、この性質を構造的に保証した（2-2の相互作用マトリクス）。
- 教訓：**「なぜこの木が面白いのか」を作り手が言葉で説明できないまま次を作ると、次も同じ理由で面白くなる保証がない。** 一番遊ばれている作の面白さの正体を、遠慮せず名指しで尋ねる（今回はこちらから尋ねる前に、素で聞かれて答えられなかったことが企画全没の引き金になった）。

### 四・一番好きな既存作が壱に違反しているときは、ゼロから発明せず「違反した一点だけ」を差し替える
- 「塗る3Dはスプラのパクリだから畜学に違反するよね？そこを直したゲームだけでも十分超えれると思う」という指摘は、それまでの「2Dでは3Dの面白さを超えられない」という行き詰まりを、**ゼロから新作を発明する問題から、既存の一番良い木の"壱に違反した一点だけ"を治す問題へ**変換した。
- 実装は `splat3d/index.html` を丸ごとフォークし、面白いエンジン（3D移動・空蹴り・🏪経済・❤️3つの緊張・世界ランキング・合成音）はそのまま継ぎ、中心動詞「インクを塗って面積%で勝つ」（＝スプラの中心フックそのもの）だけを「絡み合う5つの力で世界を作り変え、逆縁の巣を鎮める（open method）」に差し替えた。勝敗も面積%を使わず、巣まわりの purity（枯でない割合）×持続時間に変えた──**評価軸そのものがスプラの模倣にならないよう、勝利条件の数式まで一緒に差し替える**必要があった（動詞だけ変えて評価軸を面積%のまま残せば、壱違反が形を変えて残る）。
- 教訓：**「愛されている既存作が七相のどれかに違反している」と分かったとき、選択肢は"作り直す"と"捨てる"の二択ではない。違反している一点（動詞・評価軸のどちらか、または両方）を特定し、そこだけを差し替えて残りを継ぐ、という第三の道がある。** 塗る3D自体は残したまま（現行のままプレイ可能）、違反を直した版を新しい木として並べたのもこの判断の一部。

### 五・CA（セル・オートマトン）は「敵が0体・汚染が0の状態」を明示的に検証する
- 実装直後にヘッドレスChromiumで実測すると、プレイヤーが何もしないままカウントダウン明けから数秒で全4つの巣が自動的に鎮まり、勝利していた。原因は、世界生成直後は巣の周囲に「枯（BLIGHT）」セルが一つも無いため、purity（枯でない割合）の計算が最初から100%になり、巣の最初の脈動（枯を撒く周期）より先にシールタイマー（2秒の持続判定）が満了していたこと。
- 修正は、世界生成時に巣の周囲をあらかじめ72%「枯」で汚染してから開始すること。**「進行度を測る指標」を新設したら、必ず「まだ何も達成していない初期状態」でその指標がどう振る舞うかを検証する**──今回は「まだ何も浄化していない＝しかし敵の脈動もまだ来ていない」という一瞬の空白が、指標を満点に誤読させた。
- 教訓：**時間で変化する2つの力（プレイヤーの浄化とNPCの汚染）を同じ指標で測るとき、両方が0の初期状態を必ず境界条件として実測する。** 片方だけを想定してテストを書くと、"何もしていないのに勝つ"のような対称性の欠落を見落とす。

### 六・拡散確率は「1tickあたりの当たり確率」でなく「1秒あたりの期待成長数」で設計する
- 💧水・🌱草のCA拡散ルールを、直感で「1tickにつき隣接セルへ拡散する確率」として実装した（水10%・草8%）。実測すると、CAが12Hzで動き最大4方向の隣接セルへ判定するため、1秒あたりの期待成長数が水4.8個・草3.84個という指数的な値になっており、1発撃って放置するだけで数秒後にマップの離れた場所まで水浸しになる抜け道が生まれていた──畜学の核（別解の多さ）を、たった1つの安易な必勝法が潰しかけていた。
- 修正は、目標とする「1秒あたりの新規セル数」を先に決め（水0.5個/秒・草0.3個/秒）、そこから逆算して1tick・1方向あたりの確率（水1%・草0.6%）を出すこと。実測で「25m離れた場所には5秒後も水が届かない」ことを確認した。
- 教訓：**周期的に多方向へ判定する拡散・伝播ルールは、必ず「1tickの確率」でなく「1秒あたりの期待成長数（tick数×方向数×確率）」に変換してから数値を決める。** tick数や方向数を後から変えると、確率をそのままにしていても体感の速さが指数的に変わる。

### 七・検証は「周辺システム」でなく「中心動詞」を最優先で確かめる
- 公開・マージまで済ませた直後、「ブラックこっこちゃんを倒す手段がないよ、なんで塗る3Dから引き継がなかったの？」という指摘を受けた。要件定義には「武器一式は力の銃1本に統合する」と自分で書いていたのに、実装した `fireForce()` は常に地面のセル（`aimGroundCell()` の着弾点）にしか力を適用しておらず、敵に照準を合わせて撃っても地面が変化するだけだった。倒す経路は「プレイヤーが撒いた🔥/💧のセルへ敵が自分から歩いて入り、居座り続けたときだけ発動する受動的な環境ダメージ」のみで、狙って直接倒す能動的な手段が丸ごと欠けていた。
- 出荷前の検証では、CAの相互作用（延焼・消火・浄化）・巣の封鎖判定・弾薬の増減・時間切れの終端まで、実際に遊んで数値を確認していた。しかし**「照準を敵に合わせて撃ったら何が起きるか」という、遊びの中心動詞そのものを一度も検証していなかった**──周辺システム（経済・CA・終端）を検証し尽くしたことで、逆に一番基本的な入口を検証したという錯覚に陥っていた。
- 修正は `findAimEnemy()`（こっこGP/塗る3Dのロックオンと同じ、俯角を含めない水平コーン判定──俯角を含めると遠距離で内積が頭打ちになり判定できなくなるという塗る3Dの既知の不具合を最初から踏まえた設計）を新設し、`fireForce()` が地面より先に敵を探すようにした。力ごとに手応えを分けた（🔥一撃撃破・🪨二撃撃破・💧鈍化と浸水の蓄積・🌱ノックバック）。検証フックに `listEnemies()`・`unlockAll()`・`aimAt()` を足し、4力とも実際に直撃させて数値で確認した。
- 教訓：**「設計書に書いた」は「実装した」の証明にならない。出荷前の検証リストの一行目には、必ず「このゲームの一番基本の動詞（このゲームなら"狙って倒す"）を実際にやってみたか」を置く。** 周辺システムをどれだけ検証しても、中心動詞そのものを一度も検証していなければ、遊び手から見て一番の欠落を素通りする。

## 巻二・ついに ─ 判断の記録：コッコのすみか「結んで、住まわせて、ゆるす」の場合

設計（要件定義書を書く頭）と実装（コードを書く軽量な頭）を分けて建てた、初めての木の記録。

### 一・設計と実装で頭を分けるなら、仕様書は「数値と禁止事項」まで書き切る
- 今作は設計担当が十の蔵を読んで `docs/要件定義-コッコのすみか.md` を書き、実装担当（軽量な頭）がそれだけを頼りにコードを書いた。思考学が目指してきた「軽量な頭でも再現できるレシピ」の実地試験。
- 分かれ道：仕様書に何をどこまで書くか。動詞・雰囲気だけ書いて数値を任せるか、数値・禁止事項・検証の合格条件まで書き切るか。**書き切る方を選んだ**（領域半径0.62S/0.92S・出現間隔2.0秒・配点・影の初出12秒・断つまでの接触1.2秒・検証フックの関数名と引数まで）。結果、実装は仕様からほぼ迷いなく降り、one-shotで9割が正しく動いた。
- 教訓：**頭を分けるときの仕様書は「思想の共有」ではなく「数値の指定」で書く。** 思想は蔵（畜学・廿式）への参照で足りる。実装者が発明すべき余地（見た目の細部・音色）と、発明してはいけない領域（骨格・数値・禁止事項）を分けて明記する。

### 二・廿式から降ろすとき、規則は「そのまま」メカニクスにする（意訳しない）
- 分かれ道：廿式の「赦の実行は住から」「絶は対象外」を、遊びやすさのために緩める（どこからでも救出できる・絶からも戻せる）か、規則のまま実装するか。**規則のまま**を選んだ。
- 結果、「住まわせないと救えない」が防衛と成長を一本の動機に束ね（住は点数のためでなく赦の資格のため）、「幅の外は戻らない」が5秒の救出猶予に本物の緊張を与えた。規則を緩めていたら、住はただの加点、乱域はただの演出になっていた。
- 教訓：**カーネル（廿式）から降ろすときは、規則を難易度調整の対象にしない。規則が生む不自由こそが遊びの骨格になる。** 調整はスポーン間隔・速度・時間など規則の外の数値で行う。

### 三・「二つのまま参照（∿）」が既視の禁を守る芯になった
- 合体（故koro）・消去（link）と骨格を分ける一線は、廿式の ∿＝「二つのまま参照」だった。結んでも消えない・融合しない・昇格しても二軒は残ったまま上に器が浮かぶ──この一点を禁止事項として仕様書に明記したことで、実装が既存ジャンルへ滑り落ちる余地を塞げた。
- 教訓：**既視の禁は「似せない努力」より「絶対に破らない一線を一つ決める」方が守りやすい。** 本作の一線は「何も消えない・何も融合しない」。

### 四・検証フックには「状態の数」だけでなく「座標」も最初から出す
- 実装が用意した `window.__sumika.state()` は個数と状態だけを返し、座標が無かった。フック経由の検証（linkAt等）は全て通ったが、仕様の合格条件にある「実ポインタイベントでむすぶ成立」を検証する段になって、**画面のどこをなぞればよいか分からない**ことが判明。ランダムな座標でなぞる誤ったテストは12回全て失敗した。state()にこっこ・家の座標を追加してからは、最も近い二羽を狙って1回で成立した。
- 教訓：**実入力（マウス・タッチ）での検証を合格条件に入れるなら、検証フックは最初から「押すべき座標」を返す設計にする。** フックだけで完結する検証（関数を直接呼ぶ）と、実入力の検証（座標が要る）は必要な窓が違う。
- 同じ検証の過程で、目視でしか見つからない不具合も二つ出た（HUDのBEST表示がミュートボタンと重なる・住んだ二羽が家の灯を隠す）。**数値の検証が全部緑でも、スクリーンショットの目視は省略しない**（技術学の教訓と同じ形）。

### 五・湧き続ける世界では「総数の増減」は検証にならない
- 「絶で消えたら新しい認が湧く」の検証を、最初は「こっこ総数が一度減って戻る」で書いたが、通常スポーンが並走するため総数は一度も減らず、テストだけが失敗した（ゲームは正しかった）。IDの集合で「追放された個体のIDが消えた」「開始時に無かった新しいIDが現れた」を直接見る形に直して初めて、絶と継の両方を分離して実測できた。
- 教訓：**個体の出入りを検証するときは、集計値（count）でなく同一性（ID）で追う。** 集計値は並走する別システムの影響を受けて偽陰性を出す。
