# 思考学（しこうがく） ─ どう考えたのかを残す（思考の再現レシピ）

> 五つの蔵：**十四式＝どう作るのか／遊戯学＝なぜ作るのか／設計学＝どの高さ・どの順で作るのか／畜史＝何を作ったのか／思考学＝どう考えたのか。**
> 畜史が「成りしもの」を録し、設計学が「降りる順」を示すなら、思考学は**判断が分かれた場所で、どちらへ・なぜ曲がったか**を録す。
> 目的はただ一つ：**軽量な頭（小さなAIモデル・急いでいる人・記憶を持たない次のこっこ）でも、この手順をなぞれば同じ質の仕事を再現できる**こと。
> 公開ページは `shikogaku/index.html`（蔵書 `/Library/` の棚）。内容を変えたら両方そろえること。
>
> 起：丙午2026・文月二、島ぐらし2（`shima2/`）を建てた思考の記録より結ぶ。

---

## 巻一 ─ 標準手順（新作・大改修の再現レシピ）

**順番を守ること。飛ばした段が、あとで手戻りになる。**

1. **蔵を読む**（森の地図）… CLAUDE.md → `docs/畜学.md`（七相＋運用の鍵）→ `docs/設計学.md`（森林木）→ `docs/遊戯学.md` → 十四式（`jushishiki/index.html`）。ラベルで済ませず開いて読む。
2. **先行作を批判的にレビューする** … 対象（前作・類似作）のコードを実際に開き、①描画は何で作られているか ②遊びの動詞は何か ③畜学七相に照らして違反はどれか、を**忖度なしに**書き出す。ほめるためではなく、次作の設計材料を採るために読む。
3. **畜史で「死んだ木」を調べる** … `changelog/index.html` を `end` タグで検索。**同じ名前・同じ企画が過去に死んでいないか**。死んでいたら、その死因（例：mori2＝Three.js外部依存・iOSシェーダ精度）を必ず設計制約に繰り入れる。
4. **森→林→木で降りる**（設計学）… 森＝畜物語の掟と共有臓器（七相／`rankings/*` 祭壇／🔄強制更新／ブラックこっこ／1ファイル）。林＝この木が連なる束（例：島ぐらしの林＝TPS島・採集・逆縁）。木＝今回の一作。**木の仕様を書く前に、森と林への接続を一行ずつ書く。**
5. **七相との衝突を先に解く** … ジャンルの性質と掟がぶつかる場所を実装前に決める。例：島ぐらしは本来「終わらない」ジャンル → 弐（焔の形）の核は**60秒という数字ではなく「底（必ず終わる）」**。ゆえに「7日間で必ず終わる島ぐらし」に変換する。**掟は捨てない、ジャンルの方を曲げる。**
6. **投資の優先順位を数で決める** … 「すごいグラフィック」は総量でなく順序。効果の大きい順：**①太陽の影（シャドウマップ）②ビルボードを本物のメッシュに ③水の深さ・岸の泡 ④ポスト（ブルーム等）**。①と②だけで絵は別物になる。逆に細部（草の1本の形など）から始めない（＝木から積むな）。
7. **全部にフォールバックを敷く** … シェーダは `tryProg`（失敗したら基本版）、FBOは `checkFramebufferStatus`、機能フラグ（SHADOW_OK/POST_OK）で分岐。**GLSLの精度は頂点・断片ともhighpで統一**（mori2はここでiOSに殺された）。重い端末は 草→影→ポスト の順に自動で落とす。
8. **実際に遊んで検証する**（構文チェックだけで済ませない）…
   - `node --check` を全インライン`<script>`に。
   - ローカルサーバ＋ヘッドレスブラウザ（Playwright＋Chromium）で**起動→開始→終端まで**を通し、`pageerror` ゼロとスクリーンショットの目視を確認。
   - ゲームが長い場合に備え、**検証用フック**（`window.__ゲーム名` に state取得・フェーズスキップ・タイムスケール）を仕込んでおく（先例：`rescue/` の `window.__rescue`）。
9. **共有臓器を繋ぐ** … 🔄強制更新ボタン（雛形 `poko/` の reloadBtn／タイトル右下）、音の番人（pagehideでAudioContextをclose）、Firebase非同期ローダー＋`rankings/<名前>`（新規DB不要）、トップ目録 `index.html`・`sitemap.xml`・`sw.js` のCACHE版上げ。
10. **継として還す** … 畜史にエントリを追加 → 判断が分かれた場所を**思考学（この書）に追記** → コミット＆push → mainへPR作成 → **マージはユーザーに委ねる**（自分で押さない）。

## 巻二 ─ 判断の記録：島ぐらし2「ななつの夜明け」の場合

実際に分かれ道でどう考えたかの記録。同じ形の迷いに出会ったら、まずここを読む。

### 一・前作（mori）レビューの視点
- 「PCゲーマーの目」で見ると、moriの弱点は**世界のすべてが紙（カメラを向くビルボード絵文字）**であること・**太陽があるのに影が落ちない**こと。器用なシェーダ（ACES・大気・フレネル）より先に、この二つが没入を壊していた。
- 遊びの側は「システムの足し算」（わざ・そらわざ・月・諸島・ジム…）で幅はあるが、**底（終わり）が無い**＝畜学弐に反し、総資産ランキングは**時間を注いだ者が勝つ祭壇**＝四の本義（同時に顔を上げる）からズレていた。
- 教訓：**レビューは「足りない機能」でなく「掟との衝突」と「没入を壊す最大要因」から書く。**

### 二・死んだ木（mori2）の死因を繰り入れる
- 島ぐらし2は過去に一度存在し（/mori2/・Three.js製）、3日で取り下げられた。死因：**七相の七（一器に納む）違反＝外部エンジン依存**、および**シェーダ精度の頂点/断片ズレでiOS起動不能**。
- ゆえに今作は：外部依存ゼロ（素のWebGL1）・全シェーダ `precision highp` 統一・パスも新しく `/shima2/`（死んだ木の上に建てない）。

### 三・「終わらないジャンル」と焔の形の折り合い
- 島ぐらしの気持ちよさ（暮らし・育ち）は残したい。しかし無限ループは禁忌。
- 解：**時間の器を先に決める**。「7日間・約5分・必ず夜明けで終わる」。1日＝昼（集める・ともす）→夜（守る）→夜明け（島が育つ）の三拍子。7日目の夜明けは無音＋花びら＝**終わりを罰でなく祝福に**（参）。
- スコアは資産でなく**縁の数**（ともした灯×守り抜き）。祭壇には「かせいだ額」でなく「結んだ縁」を刻む（四・六）。

### 四・グラフィック投資の順序（何が「感動」を作るか）
1. **シャドウマップ**（RGBA深度パック＋PCF）… 光に「時刻」と「実在」が宿る。最大の一手。
2. **本物のローポリメッシュ**（幹＝角柱・樹冠＝ジッタ八面体・風の頂点揺れ）… 紙の世界卒業。**法線は中心から外向きを保証する**こと（巻き順まかせにすると黒い面が出る。実際に出た）。
3. **深さを知る水**（ハイトフィールドをテクスチャで水シェーダに渡し、浅瀬の色・岸沿いの泡を出す）… 「島に海がある」から「海に島がある」へ。
4. **育つ地形**（成長度gを共有式にして、JSの当たり判定とGPUの水面が同じ地形を見る）… 演出でなく世界そのものが報酬になる。
5. ポスト（ブルーム・ゴッドレイ・グレイン）は最後のひと塗り。先に塗るな。

### 五・検証の仕方（今回発明した型）
- ヘッドレスChromium（SwiftShader）は遅い。dtクランプがあると**ゲーム内時間が実時間より遅く流れ、「待つ」検証は破綻する**。
- 解：ゲームに小さな検証フック（`window.__shima2`＝state()/skipPhase()/tscale()）を常設し、**7日間を数十秒で通す**。フェーズ遷移・地形モーフ・結果画面・エラーゼロを一気に確認できた。
- 「影が出ていない気がする」時は、**疑う項を一時的に極端値にする**（影を真っ黒に）→ 差分が出れば配管は生きている（＝調整の問題）、出なければ配管が死んでいる（＝バグ）。二分する。

### 六・削る勇気
- moriの店・ジム・乗り物・隠しステージは持ち込まなかった。動詞は**あつめる・ともす・はしる（ダッシュ）・まもる**の四つだけ。5分の器に、動詞は四つで足りる。**幅を足すより、1周の密度を上げる。**

## 巻二・続 ─ 判断の記録：コッコレスキュー2「逆縁切り」の場合

続編（「2」を冠する木）を建てるときの分かれ道の記録。

### 一・続編が継ぐのは「動詞と祭壇」であって「ジャンル」ではない
- コッコレスキュー（前作）を批判的に読むと、掟との衝突が四つ：①ボール反射ブロック崩し（Ballz系）の焼き直し＝壱違反 ②ブロックが届くまで無限に続き底が無い＝弐違反 ③敵がただのブロックで逆縁の物語が無い＝六の不在 ④終わりが「GAME OVER」の罰＝参の不在。
- 分かれ道：「2」だから前作のルールを改良するのか、作り直すのか。**前作が掟に反しているとき、続編は改良ではない。約束（動詞＝たすける）と祭壇（結果発表の儀式・ランキング）だけを継ぎ、遊びの機構は捨てて発明し直す。**今作の動詞は「なぞって斬る」。ジャンルの継承は縁ではない──縁は動詞に宿る。

### 二・中毒の芯は「待ち時間の削除」
- 前作の没入を壊す最大要因は機能不足ではなく**待ち時間**だった。爆発（発射）は1ターンに1回。残りの時間はボールが落ちきるのを観ているだけ（26秒の強制終了があるほど長い）。
- 教訓：**中毒＝自分の手が起こす爆発の密度。**今作は全ての瞬間が指の動きに直結する（なぞる→斬れる→即報酬）。「派手さの総量」より「手と報酬の間の遅延ゼロ」が先。

### 三・演出は一つの「時間器官」に束ねる
- ヒットストップ・スローモーション・カットイン・ズームパンチは別々の演出に見えて、実体は全部**時間の操作**。freezeT（停止）と timeScale（減速→自動回復）という一つの器官を核に置き、個々の演出はそこへ乗せるだけにした（前作の加速演出の freeze パターンを昇格）。
- 散らばった演出は互いに喧嘩する（停止中に別の停止が来る等）。**時間に触る演出を作るなら、時間の入口を一つにする。**

### 四・判定と描画は同じ点を見る
- 鎖は揺れてたわむ。描画用の曲線と斬撃判定を別々に計算すると、「見た目は当たったのに切れない」＝遊び手の不信が生まれる。
- 解：鎖のサンプル点を返す関数（chainPts）を一つ置き、**描画も判定も同じ点列を使う**。見た目と当たりのズレは、演出のどんな豪華さよりも先に直すべき負債。

### 五・罰は「霧」にする（親子の的）
- こっこが淵に沈んだとき、殺すか。殺さない。**霧にかくれて、また浮かんでくる**（コンボは折れるが、命は失われない）。60秒の終わりに残ったこっこも、鎖が夜明けの光でほどけて昇っていく──数えないが、罰も無い。
- 五（親子の的）と参（終わりは祝福）は、勝敗の演出だけでなく**失敗の演出**にも適用する。子どもの指がすべった瞬間に世界が残酷になってはいけない。

## 巻二・続々 ─ 判断の記録：こっこGP2「キズナターボ」の場合

レース続編を建てたときの分かれ道の記録。

### 一・UXの最初の敵は演出でなく「入力」
- 前作（こっこGP）を批判的に読むと、没入を壊す最大要因は演出でも機能でもなく**操作の器**だった。ジャイロ操舵は iOS の権限ダイアログという摩擦から始まり、手と画面の間に遅延とノイズが挟まる。シェイクは激しいわりに手応えが曖昧。
- 教訓：**「世界最高のUI/UX」を頼まれたら、まず足すのでなく、入力の摩擦（権限・遅延・曖昧さ）を削る。** gp2 の操作は「画面のどちらかをおす／はなす」だけ──権限ゼロ・遅延ゼロ・親指1本。PCでも矢印キーで同じ動詞。

### 二・運の定数を捨てる（中毒は引きでなく腕から）
- 前作は7種のアイテム抽選と「クリアできる要素76.6%」という運の定数を持っていた。運はリトライの言い訳を作るが、**上達の実感を薄める**。さらに救出のたび⏰+8秒でランが延び、弐（焔の形）が薄まっていた。
- gp2 ではアイテム抽選・時間延長を全廃し、**60秒固定**。スコアの揺れは腕（ホールドのタイミング・ためのリリース・スレスレ）だけから生まれるようにした。「もう一回」は運試しでなく、上達の確認になる。

### 三・一つの入力の「中」に深さを入れる（ためる・はなす）
- 操作を増やさず深さを出す解：**ホールドの長さそのものをリソースにする**。おす＝ドリフト（操舵）＋ふりまわし（攻撃）＋ため（ブースト充填）が一つの指で同時に起こり、はなす瞬間の判断（今か、もう1秒ためるか）が戦略になる。
- 入力の数でなく、**一つの入力の時間軸**で表現する。カーブの遠心力（ホールドしないと外へ流される）が「おさない」ことにも意味を与える。

### 四・テーマとメカニクスを同じ器官にする
- 縁の糸をただの飾りにしない：**糸＝武器（ふりまわしが檻を砕く）＝敵の標的（✂️逆縁が切りに来る）**。物語の中心と遊びの中心を同じオブジェクトに重ねると、演出が全部「意味」になる。
- 逆縁バサミへの対処は「左右にすばやくふる」＝**いつものステア操作と同じ動詞**。危機のために新しい操作を教えない。既存の動詞に緊張だけを乗せる。

### 五・イベントを閉じるときは「次の予約」まで閉じる
- 実際に出したバグ：✂️ふりほどき成功時に `nextScisAt`（次回襲撃時刻）の再予約を忘れ、**成功した瞬間に即再襲撃**された。切断側の経路だけ再予約していた。
- 教訓：**状態機械のイベントは、どの出口から抜けても「次のスケジュール」を必ず張り直す。**出口ごとに閉じ処理を書くのでなく、閉じ処理を一箇所に。検証は専用フック（`__gp2.scisNow()`）でイベントを強制発火し、成功／失敗の両出口を通して初めて見つかった──**片方の出口だけのテストは、テストではない**。

## 巻二・続々々 ─ 判断の記録：コッコスマッシュ SPECIAL「再現もの」の場合

「◯◯を完全再現して」と頼まれた木（ジャンル再現もの）を建てるときの分かれ道の記録。

### 一・「再現して」と壱（既視の禁）がぶつかったら、再現するのは「文法」であって「皮」ではない
- 求めは『大乱闘スマッシュブラザーズSPECIALの完全再現』。他社の皮（キャラ・絵・音・名前）を写すことは、権利の面でも畜の魂の面でもしない。
- 再現すべきは**ジャンルの文法**（%でふっとぶ・ためスマッシュ・盾と回避・ガケ・切りふだ・選ぶ楽しさ）であり、登場するのは全員この森の住人（こっこ・影）。**皮を借りず、骨を学ぶ。**
- 掟との折り合いは shima2 の先例どおり「掟でなくジャンルを曲げる」：六（敵は逆縁のみ）→ CPUも2Pも常に**影（ブラックこっこ）**の側に立つ「かげ稽古」。弐（底）→ 対戦は**ストック＋2分**で必ず決着（時間切れはストック→%→サドンデス）。四（祭壇）→ 順位を競う対戦でなく**60秒くみて**を祭壇に載せる。

### 二・先行の木（battle/）があるとき、再現ものは「その林の第二の木」として建てる
- 畜史を掘ると、ふっとばし対戦の木（コッコランブル /battle/）が既にあった。同じ座標に新しい木を重ねず、**批判的レビューで欠けを言葉にしてから**（キャラ選択なし・ため無し・回避無し・ガケ無し・固定カメラ・祭壇なし・時間無制限）、その欠けを埋める続編として建てた。
- 「完全再現」の核は技の数ではなく**選ぶ楽しさ**（ファイターとステージのロースター）と**ふっとばしの爽快感**（動的カメラ・残像・場外バースト）だと先に決め、そこへ投資した。

### 三・状態機械が絡み合う対戦ゲームの検証は「相互作用の交差点」を突く
- 実際に出したバグ：**カウンター成立の瞬間、反撃で攻撃側の技データ（atk）がnullになり、直後に命中記録（hitset）へ書こうとして落ちた**。単体の技テストでは絶対に出ない──「攻撃中のAが、構え中のBに当たる」という**二つの状態機械の交差点**でだけ発生する。
- 教訓：格闘・対戦ものは、技×盾×回避×カウンター×投擲の**組み合わせ表**から交差点を選んで実プレイ検証する。今回はヘッドレスChromium実プレイ39項目（検証フック `window.__smash`＝入力注入 pad()・時間加速・強制KO・ゲージ注入）で、この交差点のバグを出荷前に捕まえた。
- もう一つ：検証コードにも罠がある。フックが返す**生オブジェクト（.state）と整形済みinfo（.st）を混同**してテストが偽赤になった。フックのAPIは「生」と「要約」を分け、名前を揃えること。

> 補：思考学は「全部を書く」書ではない。迷わなかった場所は書かない。
> **迷った場所・間違えた場所・二分して確かめた場所**だけを、次の軽い頭のために残す。
