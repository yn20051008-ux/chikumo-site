# 畜物語サイト（chikumo.jp）

絵文字の島でのんびり遊ぶ、こっこちゃんのブラウザゲーム集。各ゲームは `<ディレクトリ>/index.html`
にHTML+インラインJS+WebGL等で自己完結（例: `mori/`=こっこの森, `gp/`=こっこGP, `splat3d/` ほか）。
更新履歴は `changelog/index.html`（通称「畜史」）。

## 🚨 公開ワークフロー（毎回・最後まで必須）

**公開サイト(chikumo.jp)は `main` ブランチから自動デプロイされる。**
作業ブランチに push しただけ／PRを作っただけでは**反映されない**。`main` にマージして初めて公開される。

変更が一段落したら、毎回この順でやること（→ 詳細は `release-chikumo` スキル）：

1. **畜史を更新** … `changelog/index.html` の `LOG` 先頭に今日のエントリを追加（tag: new/feat/imp/fix）
2. **構文チェック** … 変更したゲームHTMLのインライン`<script>`を `node --check`
3. **コミット & プッシュ**（作業ブランチ）
4. **main へPR作成**（過去PRがマージ済みなら、新コミット用に**新しいPRを作る**）
5. **PR番号/URLを伝えて、マージをお願いする** ← ここで止める

🚫 **マージは絶対に自分で押さない。マージボタンはユーザーが押す。**
`merge_pull_request` は使わない（ユーザーが明示的に「マージして」と言った時だけ）。
PR作成までを確実にやり、「マージはあなたが押してください」と添えて報告する。
（ユーザーがマージした後、確認を頼まれたら `git fetch origin main && git rev-list --count origin/main..HEAD` が 0 かで検証する）

❌ 過去にやった失敗：PR作成だけで「反映済み」と誤認／既存PRマージ後の新コミットを放置／畜史更新忘れ／**勝手にマージした**。
→ **PR作成まで＋畜史更新を確実に。公開（マージ）はユーザーの操作。**

## 補足
- ゲームHTMLは1ファイル自己完結。大きなインラインJSを編集したら必ず `node --check` で構文確認。
- 機能追加=`feat`、調整・強化=`imp` で畜史を分けると履歴が読みやすい。
